Fundamentos: cascada, especificidad y herencia

Vamos a explicarlo como si empezaras hoy: qué regla gana, por qué y cómo verlo en las herramientas del navegador, sin resúmenes crípticos.

Cómo decide CSS (paso a paso)

1) Importancia: una regla con !important gana sobre reglas normales. Úsalo solo si no hay alternativa, porque hace el CSS difícil de mantener.

2) Especificidad: si no hay !important, gana el selector más “preciso”. Piensa en puntos: inline=1000, #id=100, .clase/[attr]=10, tipo (p, h1)=1.

3) Orden: a igualdad de especificidad, gana la regla escrita más tarde (la que está “debajo” en el archivo).

4) Herencia: algunas propiedades (como color y font-family) se heredan a los hijos; otras (margin, border) no.

Ejemplo claro (qué color gana)

/* CSS */
p { color: green; }                /* tipo = 1 */
.highlight { color: blue; }         /* clase = 10 */
#destacado { color: red; }          /* id = 100 */
p.highlight { color: purple; }      /* tipo + clase = 11 */

/* HTML */
<p id="destacado" class="highlight">Texto</p>

/* Resultado: color rojo.
   Razón: el selector por id (#destacado, 100) tiene mayor especificidad que
   .highlight (10), p (1) y p.highlight (11). */
/* Empate de especificidad: gana la última */
h1 { color: teal; }     /* 1 */
h1 { color: orange; }   /* 1 → esta está después, gana: orange */

Herencia con ejemplo

/* El color y la fuente se heredan a los hijos */
body { color: #333; font-family: system-ui; }
article { /* hereda color y font-family */ }
article strong { color: inherit; font-weight: 700; }
article small { color: #666; } /* sobrescribe solo este hijo */

Sobre !important (cuando sí y cuando no)

.notice { color: orange !important; } /* forzar color (evítalo si puedes) */
#banner .notice { color: red; }        /* esto NO gana porque !important manda */
#banner .notice { color: green !important; } /* esto sí gana: misma etiqueta, mayor especificidad + !important */

Mejor alternativa: eleva la especificidad con selectores claros o ajusta el orden de carga; deja !important para casos aislados (por ejemplo, estilos de utilidades o estados de emergencia).

Box model

Cada elemento es una caja: contenido + padding + border + margin. Con box-sizing: border-box el ancho/alto incluyen padding y borde, haciendo los cálculos más intuitivos.

.card { box-sizing: border-box; /* recomendado */ }
.box {
  width: 200px; height: 100px;
  padding: 16px; border: 2px solid #333; margin: 12px;
}
Por qué funciona: El navegador calcula el tamaño usado sumando contenido + padding + border (y separa el margin). border-box cambia la fórmula para que el ancho/alto declarados incluyan padding/borde, evitando desbordes inesperados.
  • Usa box-sizing: border-box para calcular tamaños de forma intuitiva.

Unidades y tipografía

rem escala desde la fuente raíz; em desde su padre; px es absoluto (CSS pixel); vw/vh miran el viewport. Elige según si necesitas consistencia global o escalado contextual.

html { font-size: 16px; }
h1 { font-size: 2rem; /* relativo al root */ }
p { font-size: 1em; /* relativo al parent */ }
.box { width: 50vw; height: 40vh; }
Por qué funciona: rem usa el computed del tamaño de fuente en :root; em el del elemento padre. Esto permite que la interfaz respete preferencias del usuario y que los componentes escalen de forma predecible.
  • Usa rem para tamaños consistentes; em para escalado contextual.
  • vw/vh dependen del viewport; cuidado con móviles y barras.

Colores y fondos

Usa variables como “tokens” de diseño, evita hardcodear valores y cuida el contraste para legibilidad (ideal ≥ 4.5:1). currentColor hereda el color del texto y simplifica iconos.

:root { --brand: #0d6efd; }
.btn-brand { background: var(--brand); color: #fff; }
.gradient { background: linear-gradient(135deg, #0d6efd, #6610f2); }
Por qué funciona: Las custom properties se resuelven en tiempo de cálculo y heredan; cambiar :root actualiza todo. currentColor vincula el color del icono al texto, manteniendo consistencia automáticamente.

Flexbox

Para distribuir y alinear en una sola dimensión (fila o columna). Piensa en “contenedor” y “items”: eje principal y eje cruzado.

.row { display: flex; flex-direction: row; gap: 12px; }
.row.center { justify-content: center; align-items: center; }
.grow { flex: 1; }
Por qué funciona: Al declarar display:flex creas un flex formatting context; los hijos se convierten en flex items. Las propiedades de justificación y alineación distribuyen el espacio libre en el eje principal y cruzado; gap añade separación real sin márgenes colapsables.

Grid

Para layouts bidimensionales: define columnas y filas, crea áreas y relaciones espaciales. Úsalo cuando Flexbox se quede corto.

.grid { display: grid; grid-template-columns: repeat(3, 1fr); gap: 12px; }
.span-2 { grid-column: span 2; }
Por qué funciona: display:grid crea un grid formatting context; los hijos se posicionan por líneas y áreas. La unidad fr reparte el espacio sobrante proporcionalmente, y gap define separación sin afectar el tamaño de las celdas.

Posicionamiento

Controla cómo sale un elemento del flujo normal: sticky para cabeceras pegajosas, absolute para overlays. Evita usarlo para el layout principal.

.sticky { position: sticky; top: 0; }
.abs { position: absolute; left: 10px; top: 10px; }
Por qué funciona: Los elementos posicionados crean nuevos contextos y dejan el flujo (en absolute/fixed), posicionándose respecto al contenedor relevante. sticky alterna entre flujo normal y fijo basado en el scroll.

Responsive

Adapta el diseño según espacio disponible. Media queries miran el viewport; container queries miran el tamaño del contenedor (mejor para componentes).

@media (max-width: 600px) {
  .grid { grid-template-columns: 1fr; }
}

/* Container queries (soporte moderno) */
@container (min-width: 500px) {
  .card { display: grid; grid-template-columns: 1fr 2fr; }
}
Por qué funciona: Las media queries evalúan características del viewport (ancho/alto); las container queries evalúan el tamaño del contenedor, permitiendo componentes que responden a su propio espacio.

Variables CSS (Custom properties)

Centraliza tokens (colores, espacio, tamaños) y cambia temas en runtime. Declara globales en :root y sobreescribe por componente.

:root { --space: 8px; --brand: #0d6efd; }
.btn { padding: var(--space); background: var(--brand); }
Por qué funciona: Las variables son parte del modelo de valor de CSS: heredan, se pueden leer y escribir en runtime, y se resuelven tras la cascada, lo que habilita theming dinámico sin recompilar.

Pseudo-clases y pseudo-elementos

Estados como :hover y :focus mejoran interacción; pseudo-elementos (::before, ::first-line) generan contenido decorativo o tipográfico.

a:hover { text-decoration: underline; }
input:focus { outline: 2px solid #0d6efd; }
p::first-line { font-weight: 600; }
Por qué funciona: Las pseudo-clases coinciden con estados dinámicos del DOM (hover, focus, checked) sin JavaScript; los pseudo-elementos generan cajas adicionales que se renderizan como parte del elemento.

Transiciones y animaciones

Anima lo justo: prioriza transform y opacity (GPU), respeta @media (prefers-reduced-motion) y evita loops llamativos.

.btn { transition: transform .2s ease; }
.btn:hover { transform: translateY(-2px); }
@keyframes pulse { 50% { transform: scale(1.05); } }
Por qué funciona: Transform y opacity suelen componer en la GPU (evitan layout/paint), haciendo animaciones más suaves y eficientes. Las preferencias del usuario reducen movimiento cuando está activado.

Arquitectura CSS (BEM)

Nombra por bloque–elemento–modificador para separar responsabilidades y evitar selectores profundos. Componentiza estilos y limita alcance.

.card {} /* bloque */
.card__title {} /* elemento */
.card--featured {} /* modificador */
Por qué funciona: BEM reduce la necesidad de especificidad alta y el encadenamiento profundo, disminuyendo colisiones y facilitando el reuso de componentes.
  • Nombres predecibles; evita selectores demasiado profundos.
  • Componentiza estilos; limita alcance y dependencias.

Buenas prácticas

Reglas claras para evitar dolor futuro: usa layout moderno, tokens y cuida accesibilidad y performance.

  • Prefiere layout con Flexbox/Grid sobre hacks antiguos.
  • Usa variables para temas; evita repetir valores.
  • Piensa móvil primero; optimiza fuentes/imágenes.
  • Evita !important; usa especificidad controlada.

Por qué y cuándo usar cada técnica

Guía de decisiones práctica: elige la herramienta que mejor resuelve el problema.

  • Flexbox vs Grid: usa Flexbox para alineación en una dimensión (filas/columnas); Grid para layouts en dos dimensiones (rejillas complejas). Flexbox simplifica distribución y alineación; Grid define áreas y relaciones espaciales.
  • rem vs px: rem escala con la preferencia del usuario (accesibilidad). Úsalo para tipografía y espaciamiento global; usa px cuando el pixel físico sea crítico (bordes finos).
  • Variables CSS: centralizan “tokens” de diseño (colores, espacios, tamaños) y permiten temas dinámicos sin recompilar. Útiles en diseño sistemático.
  • Media vs Container queries: media queries reaccionan al viewport; container queries al espacio real del contenedor. Prefiere container queries en componentes reutilizables.
  • box-sizing: border-box: hace que padding/border se incluyan en el tamaño total, lo que evita cálculos confusos y desbordes inesperados.

Decisiones y trade-offs

Cada opción tiene costes: documenta riesgos y el porqué de tus elecciones.

  • Resets/normalizaciones: mejoran consistencia entre navegadores, pero pueden alterar estilos nativos útiles. Aplícalos con criterio y documenta excepciones.
  • Unidades de viewport (vh): pueden fallar en móviles por barras del navegador. Considera unidades modernas (dvh, svh, lvh) cuando estén disponibles.
  • Posicionamiento absoluto: útil para overlays y badges; evita usarlo para layout principal ya que rompe el flujo y dificulta la responsividad.
  • Animaciones: añaden valor visual, pero impactan batería y performance. Respeta @media (prefers-reduced-motion) y anima propiedades que usen la GPU (transform, opacity).
  • Frameworks vs CSS nativo: frameworks aceleran el desarrollo pero añaden carga y convenciones. CSS nativo ofrece control fino; decide según contexto y equipo.

Antipatrones y errores comunes

Evita prácticas que dificultan mantenimiento y accesibilidad: mejor arquitectura que parches rápidos.

  • Abusar de !important en lugar de mejorar la arquitectura de selectores.
  • Selectores demasiado profundos que acoplan componentes y dificultan el mantenimiento.
  • Layout con float o position cuando Flexbox/Grid resuelven mejor.
  • Hardcodear colores/espacios sin variables; complica el theming.
  • No contemplar accesibilidad del foco (:focus-visible) y estados interactivos.

Ejercicios

Practica lo aprendido con retos cortos y enfocadas en conceptos clave.

  1. Maqueta una tarjeta con BEM y variables (color, espacio).
  2. Crea un layout responsive con Grid (3 columnas → 1 en móvil).
  3. Haz un header sticky y un menú flex con centrado vertical.
  4. Implementa un tema claro/oscuro usando :root y variables.

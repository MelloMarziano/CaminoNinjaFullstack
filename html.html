Fundamentos

Estructura mínima de un documento y elementos clave del <head> y <body>.

<!doctype html>
<html lang="es">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Título de la página</title>
  <meta name="description" content="Descripción concisa (50–160 caracteres)">
</head>
<body>
  <main>Contenido principal</main>
</body>
</html>
  • Usa lang correcto para i18n y SEO.
  • Define charset UTF-8 y viewport para responsive.
  • Elige un único <h1> por página, con jerarquía lógica de títulos.

Guía rápida de decisiones

Mapa rápido para elegir etiquetas y patrones correctos.

Qué usar y cuándo

  • Contenido principal único → <main>. Complementos → <aside>.
  • Bloques temáticos → <section> con encabezado. Contenido autocontenido → <article>.
  • Acciones que navegan → <a href>. Acciones en la página → <button>.
  • Lista desordenada → <ul>. Ordenada → <ol>. Definiciones → <dl>.
  • Imagen con leyenda → <figure>+<figcaption>. Responsive → <picture>.
  • Datos tabulares → <table> con <thead><tbody><tfoot>, nunca para layout.
  • Formularios accesibles → <form> + <label for> + tipos de <input>.

Buenas prácticas rápidas

  • Usa un único <h1> global; evita saltos de niveles.
  • Proporciona alt descriptivo en imágenes informativas.
  • Evita target="_blank" sin rel="noopener noreferrer".
  • En formularios, asocia cada control con su <label>.
  • Prefiere componentes nativos: <dialog>, validación HTML, inputs específicos.
  • Declara lang en <html> y charset en <head>.

Estructura semántica

HTML semántico mejora accesibilidad, SEO y mantenibilidad.

<header>Cabecera global</header>
<nav>Navegación</nav>
<main>
  <article>
    <header><h1>Título del artículo</h1></header>
    <section>Sección 1</section>
    <section>Sección 2</section>
    <footer>Metadatos</footer>
  </article>
  <aside>Contenido relacionado</aside>
</main>
<footer>Pie de página</footer>

Landmarks y reglas clave

  • header (banner) y footer (contentinfo) pueden existir a nivel global y dentro de article.
  • nav (navigation) agrupa enlaces principales; evita usarlo para pequeños grupos de links contextuales.
  • main debe aparecer una vez por página y contener el contenido principal.
  • aside (complementary) es información tangencial al contenido principal.
  • section agrupa contenido temático y debe llevar encabezado; article es autocontenido y “compartible”.
  • Mantén una jerarquía de encabezados consistente. El algoritmo de outline de HTML5 no se usa ampliamente: ajusta manualmente niveles (h1 global, h2 en secciones, etc.).

Plantillas semánticas

<!-- Esqueleto de página -->
<header><h1>Marca</h1></header>
<nav aria-label="Principal">...</nav>
<main>
  <article>
    <header>
      <h2>Título</h2>
      <time datetime="2025-01-01">1 Ene 2025</time>
    </header>
    <section><h3>Tema A</h3>...</section>
    <section><h3>Tema B</h3>...</section>
    <footer><address>Autor</address></footer>
  </article>
  <aside>Relacionado</aside>
</main>
<footer>© Sitio</footer>
<!-- Figura semántica -->
<figure>
  <img src="img.jpg" alt="Descripción">
  <figcaption>Leyenda de la imagen</figcaption>
</figure>

Checklist rápido

  • ¿Hay un único <main> y los landmarks están claros?
  • ¿Cada <section> tiene encabezado y sentido temático?
  • ¿Lo que es “compartible” está en <article> con su propio encabezado?
  • ¿Contenido tangencial está en <aside> y no en el flujo principal?
  • ¿Las figuras usan <figure> + <figcaption> y las fechas <time datetime>?
  • ¿La jerarquía de títulos es consistente (sin saltos ilógicos)?

Texto

Etiquetas de texto comunes.

<h1>Encabezado 1</h1>
<p>Párrafo con <strong>énfasis fuerte</strong> y <em>énfasis</em>.</p>
<blockquote>Cita en bloque</blockquote>
<hr> Separador
<small>Texto secundario</small> <mark>Resaltado</mark>
<sup>super</sup> y <sub>sub</sub>
<code>fragmento de código</code>
  • Usa <strong> y <em> con intención semántica.
  • Mantén una jerarquía consistente de <h1..h6>.
  • <pre> conserva espacios; <code> para inline.

Principios base de HTML

Flujo de documento, elementos block/inline y estructura del <head>.

Flujo y tipos de elementos

  • Bloque: inicia en línea nueva y agrupa contenido (p, div, section, article, header, footer).
  • En línea: fluye dentro del texto (a, span, strong, em, img, code).
  • Elige semántica antes que apariencia: usa la etiqueta que describe mejor el significado.
<p>Texto con <strong>énfasis</strong> e <a href="#">enlace</a>.</p>

Estructura del <head>

<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Título</title>
  <meta name="description" content="Resumen de la página">
  <link rel="canonical" href="https://ejemplo.com/pagina">
  <link rel="stylesheet" href="styles.css">
  <meta property="og:title" content="Título social">
  <meta property="og:type" content="website">
</head>
  • viewport para responsive; description para SEO.
  • canonical evita contenido duplicado; Open Graph mejora el compartir.

Primeros pasos: tu primera página

Estructura mínima, enlazar CSS y JavaScript, y organización de archivos.

Esqueleto y enlaces

<!doctype html>
<html lang="es">
  <head>
    <meta charset="utf-8">
    <meta name="viewport" content="width=device-width, initial-scale=1">
    <title>Mi sitio</title>
    <link rel="stylesheet" href="/css/styles.css">
  </head>
  <body>
    <main>Hola mundo</main>
    <script type="module" src="/js/app.js"></script>
  </body>
</html>

Estructura de carpetas sugerida

proyecto/
├─ index.html
├─ css/
│  └─ styles.css
└─ js/
   └─ app.js

Consejos de inicio

  • Usa type="module" para aprovechar ES Modules y evitar globals.
  • Rutas relativas: ./ dentro de carpetas; absolutas desde la raíz: /css/styles.css.
  • Valida el HTML y usa el inspector del navegador para revisar landmarks.

Tablas

  <caption>Ventas trimestrales</caption>
  <thead><tr><th scope="col">Q1</th><th scope="col">Q2</th></tr></thead>

Formularios

<form action="/signup" method="post" autocomplete="on">
  <fieldset>
    <legend>Registro</legend>
    <label for="email">Email</label>
    <input id="email" name="email" type="email" required inputmode="email">

    <label for="pass">Contraseña</label>
    <input id="pass" name="pass" type="password" minlength="8" required>

    <label for="age">Edad</label>
    <input id="age" name="age" type="number" min="0" max="120" step="1">

    <label for="country">País</label>
    <select id="country" name="country" required>
      <option value="" disabled selected>Selecciona…</option>
      <option>España</option> <option>México</option>
    </select>

    <label for="bio">Bio</label>
    <textarea id="bio" name="bio" rows="3" maxlength="200"></textarea>
  </fieldset>
  <button type="submit">Enviar</button>
</form>
  • Usa <label> con for y id coincidentes.
  • Valida con atributos nativos (required, pattern, min/max).
  • Mejora accesibilidad con aria-describedby, mensajes de error claros.

}</script>

Buenas prácticas

  • Semántica primero, estilos después; evita el “div soup”.
  • Contenido accesible, navegable por teclado y con textos alternativos.
  • Optimiza imágenes; usa srcset, sizes y loading="lazy".
  • Head limpio: títulos únicos, descripciones, canonical, metadatos sociales.
  • Mantén estructura consistente y valida tu HTML con herramientas.

Por qué y cuándo usar cada elemento

  • Etiquetas semánticas (header/nav/main/section/article): comunican significado a navegadores, lectores de pantalla y motores de búsqueda; mejoran accesibilidad y SEO. Úsalas para describir estructura real del contenido.
  • Enlace vs botón: usa <a> para navegación (cambia de URL) y <button> para acciones en la misma página (enviar formulario, abrir modal). Esto mantiene expectativas y accesibilidad.
  • Imágenes responsivas: <picture>/srcset sirven imágenes óptimas por dispositivo; reduce tiempo de carga y consumo de datos.
  • Tablas solo para datos tabulares: preservan relaciones fila/columna para tecnología asistiva; evita usarlas para layout.
  • Formularios nativos: validación y semántica integradas (required, type="email"), mejores para a11y y dispositivos móviles; complementa con JS en casos avanzados.
  • ARIA con criterio: primero elementos nativos; usa roles/atributos ARIA solo cuando no exista semántica equivalente (p. ej., patrones complejos de widgets).
  • Metadatos SEO: <title>, description, canonical y datos estructurados ayudan a la indexación y a la vista previa en redes.
  • JSON-LD sobre Microdata: separar datos estructurados del marcado visual facilita mantenimiento y es recomendado por motores de búsqueda.

Decisiones y trade-offs

  • Un solo h1 por página vs múltiples: un h1 simple mejora consistencia y SEO; múltiples h1 son válidos en HTML5 por sección, pero pueden confundir herramientas. Opta por un h1 global y jerarquía clara.
  • target="_blank": mejora UX para enlaces externos, pero conlleva riesgos; usa rel="noopener noreferrer" para seguridad.
  • dialog nativo vs modal personalizado: <dialog> aporta foco y accesibilidad built‑in; los modales custom requieren manejo cuidadoso de foco/teclado.
  • Validación nativa vs JS: la nativa es rápida y accesible; JS aporta control y mensajes personalizados, pero añade complejidad. Combínalas responsablemente.
  • Contenido embebido (iframe): útil para sandboxing, pero puede afectar performance y accesibilidad; limítalo y define títulos/contexto.
  • loading="lazy": reduce costs de carga, pero puede retrasar imágenes críticas; aplica con criterio y provee dimensiones para evitar CLS.

Antipatrones y errores comunes

  • “Div soup”: estructura sin semántica; dificulta a11y y mantenimiento.
  • Usar tablas para layout en vez de CSS; rompe semántica y responsividad.
  • Enlaces sin href o botones usados para navegar entre páginas.
  • Imágenes sin alt significativo o con alt decorativo incorrecto.
  • Omitir lang en <html> y desorden en jerarquía de encabezados.
  • Abusar de ARIA cuando existe un elemento nativo que cubre el caso.
  • Inyectar HTML sin sanitizar (riesgos de XSS) o abusar de <br> para espaciar.

Validación y herramientas

  • W3C Validator: valida sintaxis y semántica.
  • Lighthouse: accesibilidad, SEO y performance.
  • Emmet: snippets para escribir HTML más rápido.

Ejercicios

  1. Estructura una página con header, nav, main, article, aside y footer.
  2. Crea un formulario accesible de registro con validación nativa y ayuda contextual.
  3. Inserta una imagen con figure y figcaption, optimizada con srcset.
  4. Agrega metadatos SEO y JSON-LD para un artículo de blog.
  5. Usa details/summary y un dialog para contenido colapsable y modal.
Consejo: Practica con proyectos pequeños (landing, blog, portfolio) y valida siempre accesibilidad y semántica; esto te llevará a “master” rápidamente.

Teoría esencial

HTML es un lenguaje de marcado orientado a describir la estructura y significado del contenido. La capa de presentación (CSS) y la capa de comportamiento (JS) deben mantenerse separadas para mejorar mantenibilidad y accesibilidad. La semántica adecuada ayuda a los agentes de usuario (navegadores, lectores de pantalla, bots) a interpretar correctamente la información.

  • El DOM es el modelo de objetos del documento que el navegador construye a partir del HTML y que JS puede manipular.
  • La semántica define qué es cada parte del contenido (artículo, navegación, sección), no cómo se ve.
  • Accesibilidad (A11y) es parte integral del HTML: etiquetas, roles y estructura permiten que más personas usen tu sitio.
  • Performance: usa recursos optimizados (imágenes responsivas, lazy loading) y un <head> limpio.
